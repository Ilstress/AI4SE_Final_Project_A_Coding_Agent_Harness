# REFLECTION.md — 反思报告

## 一、Superpowers 技能：哪些真正发挥了作用，哪些流于形式

### 真正帮到我的技能

先说 `brainstorming`。我最初以为它只是让 AI 自行生成一堆需求，实际用下来发现完全不是这样——它会持续追问，逼着我把脑子里那些还没理清楚的假设讲出来。印象最深的一次是它问我"反馈闭环的客观信号是什么"，这个问题一下子点出了我原本的思路问题：我很可能会设计出一个"让 LLM 检查自己输出是否正确"的伪闭环。正是这次追问，把我导向了后来 Feedback Pipeline 的核心设计：7 个代码级 Generator 解析 ToolResult 的 exit_code、stdout、stderr，生成确定性的 Feedback 对象。没有这次追问，这个设计不会出现。

`test-driven-development` 这个技能，我最初的预期并不高，但在 AI 协作场景下意外有效。AI 写代码有一个常见倾向：让它写一个函数，它经常会主动附加错误处理、日志、配置选项等未被要求的内容，也就是所谓的"过度实现"。TDD 先写测试的做法恰好约束了这一点：测试只验证接口契约，AI 必须按契约精确实现，而不是自由发挥。最典型的例子是 `ActionParser`，测试定义了 4 种分类的预期行为（TextOnly 对应 str、ToolCall 已知工具对应 Action、ToolCall 未知工具对应 ParseError、格式错误对应 ParseError），AI 的实现精确对应了这 4 个分支，没有多余的"优化"。

`requesting-code-review` 采用阶段式审查，先核对是否符合 SPEC，再评估代码质量本身，这一顺序有效抑制了 AI 常见的"自信偏差"。AI 生成的代码单独来看往往显得合理，但逐条对照 SPEC 检查时，会暴露出职责归属上的错误。最典型的案例是 CLI 模块的审查：确认 `_handle_run` 函数中不包含 Tool 执行、Guardrail 判断、MainLoop 状态转移、ActionParser 解析或 Feedback 生成的逻辑，因为这些职责在 SPEC 中已明确归属给其他模块。若跳过这一步审查，CLI 很可能随着开发逐渐膨胀成职责混杂的模块。（以上这段经 AI 润色）

### 实际作用有限的技能

`using-git-worktrees` 在本项目中基本没有实际使用。PLAN 中设计了并行 worktree 的策略，但实际开发中 task 之间依赖关系较紧密，Phase 8 依赖 Phase 7，Phase 10 又依赖 Phase 8 与 9，大部分 task 实际上是串行执行的。worktree 隔离在"所有 task 均可并行"的理想场景下才有价值，但在依赖链紧密的项目中，来回切换目录、合并、清理带来的管理成本超过了它的收益。

`finishing-a-development-branch` 的流程同样被简化。由于整个项目只有一个 main 分支，未采用 PR 工作流，"是否 merge、是否开 PR、是否保留分支"这类决策实际上并未发生。这更多是项目规模较小（单人、单分支）所致，而不是这个技能本身设计上的问题。

## 二、TDD 在 AI 协作下：是阻碍还是放大器？

我的结论是放大器，但需要满足一个前提条件。

TDD 在 AI 协作中节省的不是"写测试的时间"，而是"AI 偏离预期的成本"。传统开发中 TDD 的主要价值是驱动设计、防止回归；在 AI 协作中它多了一层作用：测试很可能是 AI 唯一能够真正精确理解的规约形式。

AI 阅读 SPEC 时是在"理解"自然语言，理解就有可能出现偏差。但 AI 阅读测试代码必须精确匹配输入输出，不存在"理解偏差"的空间。`FeedbackRouter` 的 11 条路由规则就是一个例子。SPEC 中写的是"GUARDRAIL + CRITICAL 路由到 GOVERNANCE"，AI 最初的实现将所有 GUARDRAIL 反馈都路由到了 GOVERNANCE。但测试中明确规定 GUARDRAIL 加 INFO 级别应路由到 RECOVERY，AI 因此必须修正实现以匹配测试。这说明测试确实是比自然语言更精确的规约语言。

前提是测试本身必须写对。如果测试偏离了 SPEC，AI 会忠实地把错误的接口实现出来。本项目中我修正过 3 次测试（off-by-one、message_history、timeout 类型问题），每一次都是因为测试没有精确反映 SPEC 中的边界条件。

## 三、Subagent 工作流：能自主运行多久而不偏离主题？

根据我的经验，大约 2 到 3 个 task 之后就需要人工介入。

在单个 task 内，subagent 的表现比较稳定：给定清晰的 SPEC 和测试文件，它能够独立完成 50 到 150 行的实现。但一旦跨 task，上下文就会丢失。例如 T8.6（RecoveryController）和 T8.7（GovernanceController）是两个独立 task，但它们的接口实际上需要与 T8.9（FeedbackPipeline）协调一致。问题在于每个 subagent 只能看到自己那一个 task 的描述，导致 RecoveryController 和 GovernanceController 最初的接口互不一致，一个返回字符串，另一个返回枚举。

我总结出的最优 task 颗粒度是：一个 task 对应一个源文件加一个测试文件，实现规模大致在 50 到 150 行。超过 200 行的 task，例如 T10.1 主循环，subagent 容易遗漏边界条件；但拆分得过细也有问题，例如给每个 Generator 单独开一个 task，会导致测试文件的管理成本上升。

一个比较重要的发现是：subagent 的偏离往往不是随机性的错误，而是在 SPEC 未覆盖的模糊地带自主发挥的结果。减少这种偏离，单纯依靠更详细的 prompt 效果有限，更有效的方式是写出更精确的测试——测试划定了接口契约，subagent 在契约范围内可以自由发挥，但在契约边界上必须精确匹配。

## 四、SPEC 质量对实现质量的影响

这里最典型的案例是冷启动验证时暴露出的 3 处 SPEC 缺陷。

我在一个新会话中，仅提供 SPEC.md 和 PLAN.md，让 AI 实现 T5.1（Action Parser）。结果 AI 在以下几处主动停下来提问：

1. `LLMResponse` 的 `tool_calls` 字段格式是什么？——SPEC 并未定义 JSON 结构。后续补充了 `ToolCall` dataclass。
2. `task_complete` 应被分类为什么？——SPEC 表述模糊。后续明确为：解析为 ToolCall(known)，但停机逻辑交由 Main Loop 处理。
3. `ParseError.error_type` 有哪些取值？——SPEC 未覆盖此项。后续补充了 `UNKNOWN_TOOL` 和 `MALFORMED_CALL` 两个值。

这件事让我意识到一个问题：SPEC 写作者（我）和 SPEC 读者（AI）之间存在明显的信息不对称。写作者脑中往往已经有一套完整的"隐性设计"，但真正落笔的内容可能只覆盖了其中的六成左右。而无论是 AI 还是人类，要正确实现，都需要接近完整的信息。冷启动验证是我目前找到的最有效的排查手段：换一个全新的 agent，不提供额外背景，观察它在哪里受阻。受阻的位置，基本对应 SPEC 的缺陷所在。

另外一个发现是：SPEC 第 6 节的数据模型写得较为精确，字段名、类型、默认值都有明确定义，对应的 T1.1 实现几乎没有偏差。而 Feedback Pipeline（第 3.6 节）更多是概念性描述，对应的 T8.1 到 T8.9 需要反复人工修正。由此来看，数据模型描述的精确程度，对实现质量的影响明显大于概念性描述的精确程度。

## 五、我认为比较有效的 Prompt / Context 策略

第一，把测试当作规约来使用。在 task prompt 中附上测试代码所描述的预期行为，比附上 SPEC 中的自然语言描述更有效。AI 更擅长"精确匹配"而非"语义理解"。

第二，引用 SPEC 的精确条款。需要修正 AI 的实现时，引用具体段落（例如"§3.6.4 要求 Main Loop 管理 HITL 超时"），比直接要求"请把超时控制移到 Main Loop"更有效。AI 似乎天然将 SPEC 视为权威依据，引用 SPEC 可以省去围绕"这个改动是否合理"进行讨论的过程。

第三，分阶段交付。先让 AI 只实现函数签名和类型注解，审查通过后再实现具体方法体，这样可以避免 AI 在接口设计阶段过早自由发挥。

## 六、凭据与分发：迫使我想清楚了哪些原本会忽略的问题

凭据安全这一部分，让我第一次真切感受到"开发环境的便利性"与"生产环境的安全性"之间存在明显差距。开发阶段使用 `.env` 文件比较方便，但真正分发出去后，用户的新机器上并不存在这个文件。这促使我完整设计了 Credential Manager：keyring 作为主存储，cryptography 加密作为回退方案，首次运行提供 `--setup` 引导流程，日志输出中对凭据进行脱敏处理。

Docker 分发则打破了"在我自己机器上能跑"这一未经验证的假设。编写 Dockerfile 的过程中才发现，`pip install -e .` 要求 `pyproject.toml` 位于特定位置，`config.toml` 需要有默认路径，凭据也需要依赖 VOLUME 才能持久化。这些问题在本地开发阶段完全没有被考虑到。

CI/CD 中缺失 `pyyaml` 依赖，是这次教训中最直接的一个例子：本地环境中因 pytest 的传递依赖恰好带入了这个包，而 CI 的干净环境中并没有。如果没有配置 CI，这个 bug 很可能要等到第一位用户在新机器上执行 `pip install -e ".[dev]"` 时才会被发现。

## 七、如果重新做一次，我会做哪些调整

先写集成测试，再拆分 task。这次是先完成各模块的单元测试，最后才写 Mechanism Demo。如果反过来，先明确 3 个端到端 Demo 的精确预期，再据此反推各模块应有的接口签名，应该能减少模块间接口不一致的问题。

更早引入 CI。这次是 Phase 13 才配置，如果 Phase 2 就配置，pyyaml 依赖缺失和 timeout_seconds 类型不一致的问题都能更早被发现。

SPEC 中应直接给出接口签名，而不只是文字描述。多个模块的 `__init__` 参数在实现过程中经历了反复调整。如果 SPEC 一开始就写明 `def evaluate(self, action: Action) -> GuardResult`，能够明显减少返工成本。

对"可并行"这一预期不宜过高。PLAN 中标注了大量"可并行"的 task，但实际执行中大部分仍是串行完成的。将并行策略设计得过于复杂，只是增加了 PLAN 本身的复杂度，实际收益有限。

## 八、对 Superpowers 方法论的一点批判

Superpowers 这套方法论背后隐含几个假设，值得逐一检验它们在本项目中是否成立。

第一个假设是，task 可以被清晰拆分为相互独立的单元。这一假设在数据模型（T1.1）和工具层（T4.1 到 T4.3）中是成立的，但在 Feedback Pipeline（T8.1 到 T8.9）中并不成立——各模块之间接口耦合较紧，拆分为独立 task 反而导致接口不一致。

第二个假设是，subagent 在单一 task 内的表现足够可靠，人工只需承担"评审"的角色。实际情况并非如此。subagent 在 SPEC 模糊地带的"自主发挥"，往往需要人工重写，而不是微调即可解决。某些情况下，评审所需的成本甚至高于自行实现。

第三个假设是，TDD 的"红—绿—重构"循环适用于所有 task。对纯逻辑模块（如 Action Parser、规则引擎）而言，这个循环效果确实很好；但对编排类模块（Main Loop、FeedbackPipeline）而言，测试很难写，因为预期行为是多个模块交互的结果，Mock 的复杂度已接近实现本身。

结合本项目来看：第一个假设部分成立，数据模型和工具层能够被清晰拆分，但反馈管道层的耦合是客观存在的，靠拆分得更细并不能解决这一问题。第二个假设基本不成立，本项目中最有价值的修正——HITL 超时职责归属、迭代计数器——都是我发现 AI 实现有误后自行重写的结果，而不是对 AI 输出的微调。第三个假设部分成立，我最终为编排模块编写的是集成测试（Mechanism Demo）而非单元测试，这与 Superpowers 所倡导的"每个模块独立单测"有所偏离，但更符合实际需求。

我认为核心的问题在于，Superpowers 把软件工程简化为"task 分解、独立实现、测试验证"这样一条线性流程。但在实际的软件工程中，模块之间的接口协调、设计决策的权衡、边界条件的发现，往往发生在"跨 task"这一维度上。subagent 机制在这一维度上存在明显的盲区：每个 subagent 只能看到自己所负责的 task，看不到它与相邻模块之间的关系。要弥补这一盲区，依靠的是更精确的接口契约——测试与类型注解，而不是把 task 描述写得更详细。

## 九、结语

完成这个项目之后，我一直在思考一个问题：当 LLM 能够完成大部分编码工作时，工程师的价值究竟体现在哪里？

我目前的答案是，工程师的价值在于代码之外的工程判断，具体体现在三个方面。

设计判断方面，需要确定哪些机制必须以代码形式实现（护栏、反馈机制），哪些可以通过配置完成（规则、工具列表），哪些应体现在提示词中（任务描述）。LLM 能够实现每一个具体模块，但无法替代人决定模块之间的边界应划在何处。

安全判断方面，涉及凭据如何存储、危险动作如何界定、边界如何设定。LLM 能够写出 `keyring.set_password()` 这样的代码，但无法设计出完整的威胁模型。

质量判断方面，涉及什么样的测试算作"足够"（590 个测试是否充分？），什么样的限制属于"可以接受"（MVP 阶段的记忆系统不支持语义检索是否可行？）。LLM 能够编写测试，但无法判断覆盖度本身是否充分。

对我而言，这个项目的意义不在于"让 AI 帮我写了一个 harness"，而在于"我把 AI 当作工具，从零构建出了这个 harness"。二者的区别在于：前者是 AI 在做决策，后者是我在做决策，AI 负责执行。Superpowers 这套流程脚手架确实帮助我守住了 TDD、代码评审、计划制定这些基本纪律，但"应该做什么"以及"是否做对了"这两个问题，仍然需要我自己来回答。
